# Part 1:頁面清單 + 使用者動向(IA)

## 七個關鍵決定

1. 食譜/材料/模具的新增、編輯:**從彈出對話框改成獨立整頁**(不再是 Dialog)。
2. 材料/模具:**新增獨立詳細頁**,有自己的網址,顯示完整資料 + 建立者/日期
   溯源(呼應 phase 4 slice 1 已有的 `createdBy`/`lastEditedBy` 欄位)。
3. **個人頁**:暱稱設定、我的食譜、我的上傳材料/模具。
4. **這次只規劃站長角色管理頁,審核/檢舉佇列留到之後**(機制細節——幾次
   觸發、處罰階梯——都還沒拍板,現在畫頁面用不到)。
5. **食譜編輯頁裡找不到材料時,內嵌「快速新增」小表單**(只填名稱等最基本
   欄位,直接加進配方,不離開食譜編輯頁),不做「離開去 /ing/new、食譜先
   存草稿」那一套。
6. **食譜詳細頁拆成分頁籤**(基本資料/配方、營養標示與過敏原、步驟與烘焙
   設定、成本與毛利),同一個網址,籤內切換,不是分開的子網址。
7. **新增「對外營養標示卡」分享頁**:跟內部食譜詳細頁分開,只顯示營養標示、
   過敏原、保存期限等法規要求內容,**不顯示成本/毛利**,取代目前下載 PNG
   貼標籤的做法,改成給連結/QR code,配方改了網頁直接更新。

## 現有頁面(維持不變,當基準)

`/`(landing)、`/r`、`/r/:id/:name`(食譜列表+詳細,兩窗格同畫面)、
`/ings`(材料列表,要精簡欄位)、`/molds`(模具列表)、`/changelog`(更新紀錄)。

`/trash`(回收桶)頁面本身不動,但**權限語意已變**(item 1 已實作):
每人只看得到自己刪的食譜;材料/模具站長看全部、一般使用者只看自己建立的。

## 使用者動向

- **訪客瀏覽**:`/` 進入瀏覽 → 食譜/材料/模具列表+詳細頁都能看,看不到
  私人食譜、看不到編輯/刪除/新增按鈕。
- **首次登入**:`/login` 成功 → 全新帳號(`displayName` 空)導去 `/me` 醒目
  提示設暱稱;信箱密碼帳號不存在 → 導向 `/register`。
  **2026-07-10 複查拍板**:Google 建帳號時暱稱一律留空、**不抄 Google 名字**
  (那多半是真名,政策是真名/email 永不當公開顯示名稱,`callback.js` 已改)
  ——所以這個引導流程對 Google/信箱密碼兩種登入都會觸發;暱稱設定前,
  公開頁面顯示預設代稱(如「未命名烘焙師」),不顯示 email 或真名。
- **新增食譜**:`/r/new` → 基本資料;配方是**搜尋式選單**(重用材料頁現有
  名稱/廠牌/規格/分類即時搜尋邏輯,材料上千筆不能用下拉選單)→ 搜尋不到
  時選單旁「找不到?快速新增」展開小表單(只填最基本欄位)直接建材料、
  自動加進配方,不離開編輯頁、不用草稿;其餘欄位事後去材料編輯頁補 →
  選公開/私人 → 存檔導到 `/r/:id`。
  **2026-07-10 複查補**:快速新增小表單也要跑跟 `/ing/new` 同一套查重
  (名稱模糊比對),不然它就是繞過查重的後門——疑似重複時提示「已有
  相似材料:OOO,直接用它?」比新增更好(常見情境就是別人已經建過)。
  **2026-07-11 拍板:配方單位一律以克為唯一真相**——一顆蛋/水果/杯重量
  不一,只有克數能代表份量。資料層本來就只存 `items[].grams`;材料的
  `unitName`/`unitGrams`(如「顆」=55g)只作兩個輔助:①輸入快捷
  (打「3 顆」自動換算成克存入,不落地)②採購清單補充顯示
  (「蛋 8顆(440g)」,買蛋論顆)。食譜詳細頁/編輯頁展示一律克。
- **編輯自己的食譜/材料/模具**:詳細頁只有本人(材料/模具的話站長也算)
  登入時才看得到編輯/刪除按鈕(逐項 `canEdit(doc)` 判斷,取代 v3.13.0
  「按鈕照樣顯示、靠 API 擋+toast」的暫時做法)→ 進對應 `/edit` 頁存檔。
- **看別人的食譜/材料**:正常顯示但沒有編輯/刪除按鈕;可預留「檢舉」按鈕
  位置,這次不接後端邏輯。
- **新增材料/模具(查重)**:輸入名稱即時模糊比對現有資料,疑似重複跳警示
  「已有相似資料:OOO,要繼續新增還是去看它?」。
- **站長修正材料錯誤**:別人建的材料詳細頁,站長身份看得到編輯按鈕 →
  存檔後顯示「上次由站長於 YYYY-MM-DD 修正」(`lastEditedBy`/`lastEditedAt`)。
- **站長角色管理**:`/admin/users` 改角色/停用帳號,不能動內容本身。
- **個人頁自我管理**:`/me` 改暱稱/大頭貼/選填工作室資訊;「我的食譜」
  含私人可篩選;「我的材料/模具」直接跳去對應編輯頁。
- **分享營養標示卡**:`/r/:id`「成本與毛利」分頁籤旁「取得標示卡連結」→
  產生 `/label/:id/:name` 連結/QR code → 傳給客人,不用登入,不含成本/毛利,
  配方改了自動更新。

## 食譜詳細頁分頁籤結構

同一個 `/r/:id/:name` 網址,籤內切換不換網址:①基本資料/配方(份數、分類、
售價、備註、模具、配方明細)②營養標示與過敏原(即時計算,不落地)③步驟與
烘焙設定(`steps`/`bakes`/`links`)④成本與毛利(只有本人或站長看得到,也是
「取得標示卡連結」入口)。材料/模具詳細頁欄位沒那麼多,不用分頁籤。

# Part 2:視覺設計風格指南

## 這個網站的視覺矛盾,要靠風格指南解決

焙啾啾同時是**兩種截然不同的介面**:①資料密集的工作工具(材料資料庫上千筆、
成本/毛利表、營養標示 8 項數字、逐筆審核)②想給客人看的溫暖手作品牌(landing
首頁、食譜詳細頁、對外營養標示卡)。純資料儀表板風格(如 Linear)太冷、純
編輯雜誌風格(如 Kinfolk)裝不下密集表格。**風格指南要讓同一套 token 在兩種
情境都成立**,不是選邊站。

## 研究參考(搜尋世界各地優質網頁得出,附引用)

- **[Linear 的密集表格](https://www.setproduct.com/blog/data-table-ui-design)**:
  「dense yet calm」——不是縮小字級塞更多行,而是用留白+單一淡分隔線取代
  滿版邊框;hover-only 的操作對觸控/鍵盤是可及性缺陷(呼應 CLAUDE.md「手機
  輸入體驗優先」)。→ 材料資料庫、營養標示表都照這個原則做。
- **[Notion/Airtable 的資料檢視](https://support.airtable.com/docs/interface-layout-dashboard)**:
  凍結首欄、可切換檢視、緊湊密度模式而非縮字級。→ 材料頁上千筆時的瀏覽模式參考。
- **[shadcn/ui + Radix Colors 的 token 架構](https://ui.shadcn.com/docs/theming)**:
  中性色階 + 品牌色 + 語意色(成功/警告/危險)分層,每階可存取性對比都驗證
  過。→ 採用這套「token 方法論」,不是照抄配色本身。
- **[Kinfolk / Cereal Magazine 編輯風格](https://siiimple.com/cereal-magazine/)**:
  留白紀律、襯線標題+無襯線內文、克制配色、攝影當主角(「Visual Hunger」
  原則——[產品是主角,不要促銷版面干擾](https://colorlib.com/wp/bakery-websites-design/))。
  → landing 首頁、食譜詳細頁的「基本資料」分頁籤用這個調性。
- **[Toast POS / MarketMan 等餐飲成本軟體](https://pos.toasttab.com/products/recipe-costing)**:
  這才是焙啾啾功能上最接近的同類產品(不是純食譜網站)——即時更新數字、
  清楚的毛利視覺化、專業信任感。→「成本與毛利」分頁籤的調性參考,不要做得
  太可愛,這頁是店主自己看的營運數字。
- **[2026 圓角趨勢](https://www.accio.com/business/rounded_corners_design_trend)**:
  圓角在心理學上更「易處理、有親和力」,但正式/專業領域仍維持直角;
  [新的 CSS `corner-shape` 屬性](https://www.smashingmagazine.com/2026/03/beyond-border-radius-css-corner-shape-property-ui/)
  瀏覽器支援還很新,不建議這次採用。→ 焙啾啾走「適度圓角」,見下方 shape 章節。
- **[AI slop 網頁設計現象](https://www.925studios.co/blog/ai-slop-web-design-guide)**:
  第一波是紫到藍漸層+`Inter`+統一圓角卡片(2023~2025 AI 建站工具的預設輸出,
  [Lovable/v0/Bolt 這類工具「3 秒就認得出來」](https://publishd.app/blog/make-ai-built-site-not-look-ai));
  被罵到爛之後,AI 工具集體轉向**第二波「安全牌」**——米白/奶油底+赤陶暖色+
  大字級斜體襯線標題+字距拉開的副標+跑馬燈文字條。核心問題不是「用哪個顏色」,
  是**設計決定有沒有具體理由**:[空泛標語、統一留白/圓角、3 欄功能卡+頭像
  見證區塊,都是「沒有為這個產品做決定」的訊號](https://medium.com/@sahilkargutkar.sk/your-website-looks-like-ai-made-it-and-thats-becoming-a-problem-e679668ca7f4)。
  → **不套用市場研究出的「暖色食品網站公式」,延伸焙啾啾現有的、已經有具體
  理由的配色**(見下)。

## 現有系統已經是對的起點(不是從零設計)

`src/index.css` 裡焙啾啾**已經有**一套具體、有理由的視覺系統,不是
Tailwind 預設值:

```
--color-paper:      #FCFBF7   米白底
--color-paper-deep:  #F4F1E7   深一階米白(區塊底)
--color-ink:         #1B1813   深咖啡黑(主文字)
--color-ink-soft:    #6B6455   次要文字
--color-line:        #E6E1D4   分隔線
--color-yolk:        #C98A12   蛋黃金(強調色)
--color-yolk-soft:   #F6EDD8   蛋黃金淡底
--color-warn:        #B4452C   警示磚紅
--color-ok:          #4A7040   成功墨綠
--font-serif: "Noto Serif TC"   襯線(現有用在營養標示標題等)
--font-sans:  "Noto Sans TC"    無襯線(介面主力)
--font-mono:  "IBM Plex Mono"   數字等寬(成本表/營養表已經在用)
```

`yolk`(蛋黃金)這個命名跟色值特別值得保留——它不是「因為食品網站流行暖色」
選出來的,是直接扣「蛋」這個烘焙核心材料,具體、有理由,這正是 AI slop
研究強調的「避免通用公式」的反面案例:**這套系統本來就沒有 AI 感**,大改版
的任務是**延伸它、讓它撐得起新頁面**,不是拿市場研究的「暖色食品網站公式」
去替換掉。

## 字體(Typography)——延伸現有 serif/sans/mono 三件套

- **維持三件套分工不變**:`Noto Sans TC` 介面主力(表單、表格、按鈕、側欄)、
  `Noto Serif TC` 品牌時刻(landing 大標、食譜名稱、營養標示標題——現有
  `.nlabel` 已經這樣用)、`IBM Plex Mono` 數字專用(成本/克數/百分比,
  `font-variant-numeric: tabular-nums` 現有 `.ltable .num` 已經在用,新頁面
  的數字都比照)。**不新增 `Inter`**——現有系統沒有它,加了反而是往「AI 預設
  字體」靠攏,Noto Sans TC 本身對拉丁字母/數字的呈現已經夠用。
- **拉丁字母/數字備援**:`Noto Sans TC` 字串補上 `-apple-system, 'PingFang TC',
  'Heiti TC'` 備援,涵蓋沒裝 Noto 字重的裝置。
- **字級與行高**:中文行高抓 1.6~1.75(現有 15px 基準內文,大改版新頁面
  維持這個下限,行動裝置廚房現場光線不佳更不能再小)。字級尺度沿用現有
  簡單等比,不用另開一套。
- 中文避免全篇左右對齊(justify)——CJK 對齊容易產生字距怪異的縫隙。
- **不要**:標題斜體+字距拉開的副標(現有系統的 `letter-spacing: .15em` 用在
  `.nlabel .title` 這種法規標題上是合理的強調,但**不要**擴大到一般內文副標
  ——那正是 AI slop 第二波的特徵之一,見下方避雷清單。

## 配色(Color)——延伸現有 paper/ink/yolk,不引入新色域

- **背景/文字/強調色維持現有 token**,不換成市場研究的暖白/赤陶組合——
  `paper`/`ink`/`yolk` 已經是同一個邏輯(米白底+深咖啡文字+蛋黃強調),
  新頁面全部沿用,不新增第二套配色。
- **語意色也維持現有**:`warn`(磚紅,私人/警示)、`ok`(墨綠,成功/公開)——
  這兩色現有的私人 🔒 chip、公開 chip 已經在用,新頁面(材料/模具詳細頁的
  站長修正標記、`canEdit` 判斷後的狀態提示)直接沿用,不用另外挑色。
- **新頁面需要補的,是現有色階不夠用的地方**,不是換掉現有色階:
  - 中性階目前只有 `ink`/`ink-soft`/`line` 三階,材料資料庫的密集表格
    (凍結列、hover 狀態、選取列背景)可能需要在 `paper` 到 `ink` 之間再補
    1~2 個中間階,用色相同色溫延伸,不引入新色相。
  - `yolk` 目前只有 `yolk`/`yolk-soft` 兩階,新增的按鈕 hover/active 狀態
    可能需要 `yolk-deep`(更深一階的蛋黃金做 hover),同樣是延伸不是新增色相。
- **深色模式**:目前整站沒有 dark mode,這次不強制做;如果之後要加,
  用同一個「咖啡色溫」邏輯反轉(深咖啡底+米白文字),不是另開一套冷色系
  深色主題,那樣會讓亮/暗兩個模式看起來像兩個不同品牌。列為待拍板。

## 圓角/稜角(Shape)

不走兩個極端(全直角太生硬、全大圓角太軟不夠專業):

- **小型元件**(Chip、輸入框、小按鈕):`--radius-sm` ≈ 6px。
- **卡片/對話框/新頁面的內容區塊**:`--radius-md` ≈ 10~12px。
- **大型視覺區塊**(landing 首頁圖片卡、食譜大圖):`--radius-lg` ≈ 16px,
  這裡可以更明顯,呼應「品牌時刻」要更柔和有溫度。
- **標籤/狀態徽章**(公開/私人/分類 chip):維持現有的**膠囊形**(pill,
  `--radius-pill` = 999px)——這是現有 Chip 元件已經在用的形狀,適合掃視辨識,
  不用改。
- 資料表格本身(材料清單、營養標示表)**不要**加圓角容器套一層卡片——直接
  融入頁面背景,用留白+單一分隔線分段(呼應 Linear 的作法),圓角留給
  「容器層」而不是表格列本身。

## 簽名系統:帳本規線(2026-07-11 第二輪複查定調)

**焙啾啾的視覺簽名是「帳本規線」,不是配色。** 這套「粗黑規線=正式資料」
語彙源自台灣包裝營養標示+紙本帳冊,是 AI 預設生不出來的具體識別——
`src/index.css` 的 `.nlabel` 本來就註記「簽名元素」,現在把散落各元件的
用法定成全站規則:

| 線寬 | 用在 | 現有實例 |
|---|---|---|
| 8px | 法規標示標題(最重的一級) | `.nlabel h4` 底線 |
| 6px | 正式對話框標題 | `Dialog.jsx` 標題底線 |
| 4px | 法規區塊內部分層 | `.nlabel .meta` |
| 3px | 頁面主標題 | 各頁 `h2` 的 `border-b-[3px] border-ink` |
| 2px | 區段小標、表格表頭底線、total 列頂線 | `.ltable th`、`tr.total` |
| 1px `--color-line` | 一般列分隔 | `.ltable td` |

**語意規則:線越粗=資料越正式**(法規標示>頁面標題>區段>一般列)。
新頁面一律沿用這個分級做視覺層次,**不用陰影、卡片框、漸層**。
搭配元素:等寬數字(IBM Plex Mono+`tabular-nums`)、Dialog 粗框直角
(帳冊蓋印意象)、Chip 膠囊(印章感)。

⚠ **施工警告**:不要「圖乾淨」把粗規線改細、或用 shadow 卡片取代規線
分層——那會直接滑回 AI 預設外觀(見下方避雷清單第一條)。

## 密集資料 vs 編輯感內容,兩種版面邏輯

| | 密集資料頁(材料/模具列表、成本表、營養標示表) | 編輯感內容頁(landing、食譜「基本資料」分頁籤) |
|---|---|---|
| 字體 | 全無襯線(Noto Sans TC) | 標題可疊 Noto Serif TC |
| 留白 | 緊湊但不擁擠(留白服務效率) | 大量留白(留白服務氛圍) |
| 分隔 | 單一淡分隔線,不加卡片框 | 可以用大圖/色塊分區 |
| 互動 | 操作按鈕永遠可見(不靠 hover),觸控友善 | 互動元素少,以內容/攝影為主角 |
| 圓角 | 小(6px 級),表格本身不套圓角容器 | 可以到 16px 級,更柔和 |

**成本與毛利分頁籤**特別歸類到「密集資料」這一側但語氣要更像 Toast/MarketMan
這類專業營運工具——清楚的數字、不用太多裝飾,因為這頁是店主自己看的營運
數字,不是給客人看的品牌內容。

**對外營養標示卡**(`/label/:id`)則是特殊情況:內容是法規表格(不能過度
風格化,格式要維持營養標示的可辨識性),但外層可以包一層暖色調的編輯感
邊框/留白,不要讓它看起來像原始 HTML 表格丟在白頁上。

## 響應式:同一套風格,不是兩套設計

**不做「桌面版」「手機版」兩套視覺**——token(字體家族、配色、圓角尺度)全站
統一,只有**版面編排**跟著螢幕寬度變,原因:兩套視覺等於雙倍維護成本,而且
CLAUDE.md 已經確認「工作室現場常用手機」,手機不是次要平台,沒理由風格降級。

真正該隨尺寸調整的是這三件事:

1. **雙欄→單欄**:食譜/材料/模具現有「側欄清單+右側詳細面板」雙窗格模式
   (桌面並排顯示)在手機寬度**改成單欄、點列表項目才切換到詳細頁**——
   這不是縮小雙欄塞進手機,是完全不同的導覽邏輯。剛好呼應「材料/模具有
   自己的獨立網址」,每頁本來就有自己的路由,手機上「列表→詳細→返回」
   用瀏覽器返回鍵/現有 hash 路由就對,不用另外做導覽元件。
2. **密集表格→卡片式清單**:材料資料庫這種多欄表格,窄螢幕硬擠會被迫水平
   捲動(體驗差,hover-only 操作對觸控是可及性缺陷,多欄橫向捲動同樣是
   觸控痛點)。手機寬度建議把「列」轉成「卡片」(每筆資料直式排列關鍵欄位),
   不是同一個表格縮字級硬塞。
3. **觸控目標放大,不是留白縮小**:手機上按鈕/可點擊區域要 ≥44px,但這不
   代表要犧牲留白配合——版面編排(欄數、字級)可以緊湊,觸控目標本身的
   尺寸不能跟著縮。資訊密度可以隨螢幕調整,人體工學尺寸(觸控熱區)不行。

**字級不做「手機再縮小」**:CJK 可讀最小字級是絕對值(15~16px 內文下限),
不是「桌面 16px、手機再降到 13px」這種相對縮放——手機握持距離也近,實際
視角字級需求跟桌面相差不大,反而行距/留白這些「呼吸空間」可以在窄螢幕
適度壓縮(不是壓字級)。

**新增/編輯整頁表單**:桌面可以考慮多欄並排(如左邊基本資料、右邊配方即時
預覽),但手機上這類表單天生就該是單欄——不用特別為手機重新設計版面,只是
要注意分步驟(基本資料→配方→步驟→標示)在手機上更重要,一次塞一整頁大
表單在小螢幕上使用體驗會很差。

## AI slop 避雷清單

2026 年網頁設計最大的審美疲勞來源:AI 建站工具(Lovable/v0/Bolt 這類)輸出
高度收斂,同類網站「3 秒就認得出來」,使用者一看到這些特徵就想關掉。核心
成因不是某個顏色/字體本身不好,是**設計決定沒有具體理由,直接套用訓練資料
裡最常見的樣板**。大改版時逐項對照,出現以下任一項就代表決定不夠具體,
要重做:

- ❌ **暖奶油底+襯線標題+赤陶強調=當前 AI 產出的預設外觀第一名**
  (2026-07-11 第二輪複查補)——焙啾啾配色跟這個族群同源,靠四件事拉開:
  ①強調色是蛋黃金 `#C98A12`(色相偏黃,不是赤陶紅橘)②帳本規線簽名系統
  (見上方章節)③等寬數字的資料感 ④landing/詳細頁展示現算真資料不是
  氣氛圖。**這四件事任何一個被稀釋,整站就退回「又一個 AI 生的暖色食品
  網站」**——施工時每個新頁面都要對照。
- ❌ 標題斜體+字距拉開的副標(第二波 AI 審美的標記性特徵)
- ❌ 跑馬燈/跑馬式文字條(用途不明的裝飾性動態文字)
- ❌ 全站統一 `border-radius`(每個元件都同一個圓角值)——焙啾啾已經是分級
  (6/10/16/pill),保持分級,不要之後施工時圖方便統一成一個值
- ❌ 空泛標語(「打造未來」這類講不出具體是在做什麼的句子)——landing 頁
  文案要具體講「這站算的是真的成本/真的營養標示」,不是抽象願景詞
- ❌ 3 欄式功能介紹卡+假頭像見證區塊——現有 landing 已經用「真食譜卡片
  現算成本/毛利/過敏原」取代這套(v3.5.0),大改版延續這個「展示真資料
  不展示假裝飾」的原則
- ❌ Stripe 藍/紫漸層卡片、玻璃霧面(glassmorphism)背景——焙啾啾整體是
  實色+分隔線的邏輯(呼應 Linear 密集表格章節),不需要漸層/模糊層
- ❌ `Inter` 當唯一無襯線字——焙啾啾用 Noto Sans TC 為主,天生不是 AI 預設
  組合,維持這個差異
- ✅ 具體理由的檢驗標準:每個設計決定都要能回答「為什麼是這樣」——蛋黃金
  強調色因為蛋是烘焙材料、數字用等寬字因為成本表要對齊、密集表格不加卡片
  框因為要塞更多真實資料、營養標示卡外層包編輯感留白因為要跟店主自己看的
  成本頁區分開來給客人看。講不出具體理由的裝飾,一律不加。
