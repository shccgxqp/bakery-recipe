# 產品路線圖(需求盤點中,尚未實作)

> 建立:2026-07-09。狀態標記:🔴 需求未定 / 🟡 需求大致清楚 / 🟢 可直接動工。
> 每項列出「待拍板的問題」——這些問題有答案,該項才算需求確定。

## 總體目標(第 4 項的願景,統領全部)

打造「**分享烘焙食譜 + 個人工作室營業用 + 營養/過敏原標示**」的網站:
使用者能建立自己的食譜、填寫營養標示、標註過敏原;工作室能算成本毛利。
從「個人工具」轉型為「多使用者平台」。

## 建議實作順序與相依關係

```
7 版本更新頁(小,隨時可做)
6 網址路由(基礎建設,越早做越省事)──┐
5 食譜編輯 UI 重造(不依賴其他項)      ├─→ 2 帳號系統(平台化的分水嶺)─→ 4 平台化大改造(公開投稿/警告系統)─→ 3 付費功能
1 拍照快照 OCR(可先做單人版)────────┘
```

---

## 1. 🟡 新增材料:拍照快照 → 自動填營養標示/成分/過敏原

拍包裝的營養標示表 → 自動辨識 8 項營養數值、成分欄文字 → 預填表單、
自動勾選過敏原(從成分文字比對 11 類關鍵字)→ 人工確認後儲存。

- **技術方向**:辨識用 AI 視覺 API(對台灣包裝的中文直式/橫式表格,傳統 OCR 準確率差,
  AI 模型效果好很多)。走 Vercel function 轉發,金鑰放環境變數。
- **成本**:每次辨識約 NT$0.1~0.5(依模型),單人用量可忽略;開放多人後要限流。
- **照片本身**:辨識完即丟,「不」存資料庫(M0 只有 512MB;未來要存照片得用外部圖床)。
- **待拍板**:
  - 照片要不要保留當佐證?(建議:第一版不保留,辨識完即丟)
  - 手機拍照為主?(材料頁在手機上要有「＋拍照新增」入口)

## 2. 🔴 帳號系統:Google 快速登入

Google OAuth 登入;使用者可投稿個人食譜,選擇公開或私人。
**這是整個平台化的分水嶺**,動到資料架構(所有資料加 `ownerId`)、
API 權限模型(從單一密碼變成 per-user 授權)、UI(個人頁)。

- **技術方向**:Auth.js(Vercel 上的標準做法)或 Firebase Auth/Clerk 免費層。
  現有的單一編輯密碼制在帳號系統上線後轉為「站長權限」。
- **schema 已預留**:公開參考資料與私人營運資料欄位已分離(見 db-schema.md),
  屆時私人欄位拆出去加 `ownerId` 即可。
- **待拍板**:
  - 只開 Google 登入?(建議:是,一種就夠,少一半麻煩)
  - 訪客(未登入)能看什麼?(建議:公開食譜+材料資料庫全部可看,編輯才要登入)
  - 私人食譜存量限制?(M0 只有 512MB,要防止被塞爆)
  - 你自己的 20 道食譜屆時預設公開?

## 3. 🔴 付費功能(最後才做——「功能通通齊全才可以實現」)

候選付費點:毛利計算(工作室營業功能)、過敏品通知。

- **相依**:2(要有帳號才知道誰付了錢)、金流(台灣:綠界/藍新/Stripe)。
- **原則(使用者原話)**:沒本事的應用程式沒資格收錢——全部功能穩定後才啟動。
- **待拍板**:免費/付費的分界線;訂閱制或買斷;過敏品通知的具體行為
  (是「我對 X 過敏,幫我標記所有含 X 的食譜」?)。

## 4. 🔴 網頁平台化大改造

視情況大幅更新介面;開放使用者自填營養標示,搭配**防亂標警告系統**:
資料被通報錯誤達一定次數 → 處罰(標示降信任度/隱藏/禁言)。

- **相依**:2(要有帳號才能通報與處罰)。
- **信任機制草案**:每筆營養資料保留「來源」(包裝抄錄/官方資料庫/使用者自填)+
  通報按鈕;通報數過門檻自動下架待審。站長(你)是最終仲裁。
- **待拍板**:通報幾次觸發?處罰階梯?自填資料要不要先審後上?
  官方(你維護的)資料與使用者資料視覺上如何區分?

## 5. 🟡 新增食譜/編輯頁 UI 重造

目前的編輯對話框欄位越來越多,要重新思考「好輸入好填寫」。

- **方向候選**:對話框改整頁編輯器;分步驟(基本資料→配方→步驟→標示);
  配方輸入支援連續快速輸入(打名稱+Enter 直接下一項)、常用材料快選;
  手機輸入體驗優先(工作室現場常用手機)。
- **待拍板**:你實際輸入食譜時最卡的是哪一步?(這個答案決定重造方向,
  建議你下次新增一道食譜時記下最煩的三件事)

## 6. 🟡 網址與 API 優化:每頁專屬網址

需求:食譜有專屬網址(分享連結直達)、個人帳戶有專屬頁面。

- **React SPA vs 專屬網址——結論:不衝突,兩個都要。**
  React「切換頁面」和「每頁有網址」是兩件獨立的事:加上前端路由(react-router)後,
  依然是 React SPA(切換一樣快、不重新載入),但每道食譜有 `/recipe/黑糖奶油棒`
  這樣的真實網址,可分享、可加書籤、上一頁下一頁正常。**繼續用 React,加路由即可。**
- SPA 唯一的真缺點是 **SEO**(搜尋引擎收錄較差)——這在「公開材料資料庫要被烘焙業者
  搜到」時才變成問題,解法是屆時把主站搬到 Vercel 做預渲染(網址結構不變,先把路由
  做對,以後搬家無痛)。
- **附帶決定**:GitHub Pages 對 SPA 路由要加 404 轉址 hack,做這項時建議順勢把
  主網址換成 Vercel(API 同源還省 CORS),GitHub Pages 保留轉址。
- **待拍板**:網址用食譜名稱(中文網址,分享好看)還是 id(穩定但醜)?
  (建議:`/r/<短id>/<名稱>`,名稱只是裝飾、id 定位,改名連結也不斷)

## 7. 🟢 版本更新專屬頁面(隨時可做)

面向使用者的「更新紀錄」頁:只寫「優化了什麼、追加了什麼功能」,不寫技術細節。

- **做法**:資料源就用 CHANGELOG.md,但每條分「使用者可見」與「技術」兩層,

- **相依**:無(可以馬上做);若第 6 項路由先做好,這頁就有專屬網址 `/changelog`。
